🧠 Od Prompt Engineering do Context Engineering

  • 2023: „Jak napisać lepszy prompt?"
  • 2024: „Jak zbudować system promptów z rolami i szablonami?"
  • 2026: „Jak dostarczyć modelowi właściwy kontekst – narzędzia, dane, pamięć?"

🎯 Context Engineering = sztuka budowania całego środowiska, w którym model pracuje

Intro:
„W 2023 cały świat mówił o prompt engineering — jak napisać lepszy prompt. W 2024 przeszliśmy do systemów promptów z rolami i szablonami. Teraz, w 2026, to za mało. Modele są na tyle mądre, że kluczowe nie jest JAK pytasz, ale CO im dajesz."

Storytelling:
🔹 „Wyobraźcie sobie nowego pracownika. Opcja A: dajecie mu lepszą instrukcję — 'bądź dokładny, odpowiadaj po polsku'. Opcja B: dajecie mu dostęp do CRM-a, bazy wiedzy, systemu ticketowego i mówicie: 'użyj tego, co potrzebujesz'. Która opcja daje lepsze wyniki? Oczywiście B. To jest Context Engineering."
🔹 „Inny przykład: lekarz. Czy lepszy prompt to 'bądź dokładny'? Nie. Lepszy kontekst to: wyniki badań, historia choroby, aktualne leki. To robi różnicę."

Aktywizacja:
🔹 Zapytaj: „Kto z Was pisał kiedykolwiek system prompt? Co w nim było?"
🔹 Zbierz 2-3 odpowiedzi — to pokaże, na jakim poziomie jest grupa.

Przejście:
„Zobaczmy, z jakich warstw składa się kontekst modelu — to fundament, na którym budujemy resztę dnia."

🧩 Co składa się na kontekst modelu?

📋 System Prompt

Rola, zasady, ograniczenia

🔧 Tools (Narzędzia)

API, bazy danych, wyszukiwarki

📚 RAG / Wiedza

Dokumenty, procedury, baza wiedzy

💾 Pamięć

Historia rozmowy, stan zadania

Wyjaśnienie:
🔹 Pokaż każdy element jako „warstwę" kontekstu. Odsłaniaj fragmenty po kolei.
🔹 System Prompt = kim jest model, jakie ma zasady.
🔹 Tools = co model MOŻE zrobić: API, bazy danych, funkcje.
🔹 RAG/Wiedza = dokumenty, procedury dopasowane do pytania.
🔹 Pamięć = historia rozmowy, preferencje użytkownika.

Storytelling:
🔹 „Pomyślcie o przygotowaniu sali konferencyjnej: System prompt = agenda spotkania. Tools = laptop i projektor. RAG = materiały na stole. Memory = notatki z poprzedniego spotkania. Im lepiej przygotowana sala, tym lepsze spotkanie."

Aktywizacja:
🔹 Zapytaj: „Gdybyście budowali chatbota dla waszej firmy — które z tych 4 warstw byłyby najważniejsze?"
🔹 Zbierz odpowiedzi — zazwyczaj padają Tools lub RAG.

Przejście:
„Przyjrzyjmy się dwóm konkretnym technikom: Tool Calling i Structured Outputs — to jest to, co odróżnia nowoczesne aplikacje AI od prostych chatbotów."

🔧 Tool Calling – model używa narzędzi

Model nie „wie" wszystkiego – ale może wywołać funkcję, żeby się dowiedzieć.


tools = [{
  "type": "function",
  "function": {
    "name": "classify_ticket",
    "description": "Classify an IT ticket",
    "parameters": {
      "properties": {
        "category": {"enum": ["Hardware","Network","Software","Account"]},
        "confidence": {"type": "number"},
        "reasoning": {"type": "string"}
}}}}]

🤖 Model widzi narzędzia → decyduje, które wywołać → zwraca structured output

Intro:
„Dwa kluczowe pojęcia nowoczesnego AI: Tool Calling — model SAM decyduje, które narzędzie użyć. Popatrzcie na ten JSON — to definicja 'narzędzia', które model może wywołać."

Storytelling:
🔹 „Wyobraźcie sobie nowego pracownika. Zamiast mówić mu 'wygoogluj to sam', dajecie mu listę narzędzi: 'tu masz CRM, tu bazę wiedzy, tu system ticketowy'. Model działa tak samo — widzi, jakie narzędzia ma do dyspozycji, i sam decyduje, którego użyć."

Wyjaśnienie:
🔹 Pokaż kod z slajdu — to definicja narzędzia w formacie JSON.
🔹 Kluczowy punkt: w 2023 musieliśmy sami parsować output modelu. W 2026 model sam zwraca dane w strukturze, którą mu zdefiniowaliśmy.
🔹 Nie wchodź zbyt głęboko w implementację — zobaczymy to w praktyce w notebooku agenta.

Przejście:
„A co model zwraca? Zobaczmy Structured Outputs — koniec z parsowaniem tekstu."

📐 Structured Outputs – JSON mode

❌ 2023


Prompt: "Classify this ticket"

Output: "I would classify this
as Network because..."

→ Parsowanie tekstu, regex...

✅ 2026


{
  "category": "Network",
  "confidence": 0.85,
  "reasoning": "Printer via IP"
}

→ Gotowy JSON, zero parsowania

Intro:
„Porównajmy stare i nowe podejście. Po lewej: 2023, model gadał jak chciał. Po prawej: 2026, czysty JSON."

Storytelling:
🔹 „Pamiętacie regex? Kto kiedyś parsował odpowiedź LLM-a, żeby wyciągnąć jedną wartość? W 2023 to była codzienność — model zwracał tekst, a Ty musiałeś go 'obierać' jak cebulę."
🔹 „To jak formularz urzędowy vs list. List możesz napisać jak chcesz — pięknie, ale trudno go automatycznie przetworzyć. Formularz ma pola: imię, data, kwota. Model ze Structured Output zachowuje się jak pracownik wypełniający formularz."

Wyjaśnienie:
🔹 Pokaż oba przykłady side-by-side.
🔹 Powiązanie z ćwiczeniem: „W naszym notebooku z agentem Pythonowym zobaczymy to w praktyce — model zwraca JSON z kategorią, pewnością i uzasadnieniem."

Przejście:
„OK, wiemy jak model dostaje narzędzia i zwraca strukturę. Ale jest jeszcze kilka parametrów, które kontrolujemy — zobaczmy je."

🌡️ Parametry modelu – co kontrolujemy?

  • Temperature – 0 = deterministic, 1.5 = kreatywny chaos
    Klasyfikacja = 0, brainstorm = 1.0+
  • Max tokens – limit długości odpowiedzi
    Krótka odpowiedź = 100, esej = 4000
  • Context window – ile tekstu model „pamięta"
    GPT-4o: 128K, Claude: 200K, Gemini: 2M tokenów
  • System prompt – „osobowość" i reguły modelu
    Kim jest, jak się zachowuje, czego nie wolno

Intro:
„Mamy kilka 'pokręteł', którymi sterujemy modelem. Najważniejsze to temperatura i context window."

Storytelling:
🔹 „Temperature to jak pokrętło kreatywności. Na 0 model jest urzędnikiem — zawsze ta sama odpowiedź. Na 1.5 jest artystą — każda odpowiedź inna, czasem genialna, czasem szalona."
🔹 „Context window to jak biurko modelu — ile dokumentów może mieć otwartych naraz. GPT-3 miał małe biurko (4K tokenów). Gemini 2.5 ma cały magazyn (2M tokenów)."

Aktywizacja:
🔹 „Jaką temperaturę ustawilibyście do klasyfikacji zgłoszeń?" (0 — chcemy powtarzalność)
🔹 „A do pisania kreatywnych opisów produktów?" (0.7-1.0)

Wskazówki do moderowania:
🔹 Pokażecie to za chwilę w Ollama — ta sama prośba na temperature 0 vs 1.5.
🔹 Jeśli ktoś pyta o top_p, frequency_penalty — powiedz: „Zaawansowane parametry, temperature wystarczy w 90% przypadków."

Przejście:
„Teraz zobaczmy, jakie modele mamy do dyspozycji — i czym się różnią."

🧱 Typy modeli językowych

  • Foundation model: ogólny, ogromny, bez specjalizacji (np. GPT-4o, Claude, Gemini)
  • Fine-tuned: ten sam model, ale podciągnięty pod konkretny temat (np. model do analizy CV)
  • RAG: najpierw szuka informacji w dokumentach, potem odpowiada

Intro:
„Nie wszystkie modele są takie same. Zobaczmy 3 główne typy i kiedy którego użyć."

Storytelling:
🔹 Foundation = encyklopedyczny student — wie dużo, ale ogólnie.
🔹 Fine-tuned = specjalista z wieloletnim doświadczeniem w jednej branży.
🔹 RAG = konsultant z dostępem do dokumentacji — sprawdza zanim odpowie.

Aktywizacja:
🔹 Zapytaj: „Który typ najbardziej pasowałby do waszej firmy?"

📊 Kiedy użyć którego?

  • Foundation: kreatywne zadania, testowanie, „pogadaj z AI"
  • Fine-tuned: dokładność, konkretne zadania, np. CV, analiza logów
  • RAG: chatbot firmowy, AI z dostępem do dokumentacji

Wyjaśnienie:
🔹 Jeśli masz swoje dane treningowe — fine-tune.
🔹 Jeśli masz swoje źródła/dokumentację — RAG.
🔹 Jeśli chcesz pogadać o sensie życia — foundation.

Wskazówki do moderowania:
🔹 Większość firm zaczyna od RAG — bo nie wymaga trenowania modelu, tylko bazy wiedzy.
🔹 Fine-tuning jest droższy i trudniejszy, ale daje najlepsze wyniki na specjalistycznych danych.

Przejście:
„Sprawdźmy, czy rozróżniacie te typy — czas na szybki quiz!"

🎨 Vibe Coding — budujemy razem!

AI pisze kod. Wy decydujecie co.

Format: 4 rundy decyzji

Każda runda → głosujecie reakcjami Zoom → trener promptuje AI

Na koniec → działająca aplikacja zbudowana w ~20 minut

🛠️ Narzędzie: Claude Code / Cursor (trener na ekranie)

Intro:
„Vibe coding to najgorętszy temat w tech w 2026 — opisujesz co chcesz, AI pisze kod. Ale zamiast ja sam decydował — wy będziecie sterować. W 4 rundach zagłosujecie, co budujemy, jaki framework, jakie funkcje. Ja tylko promptuję AI."

Jak poprowadzić:
🔹 Otwórz Claude Code lub Cursor na projektorze (screen share).
🔹 Po każdej rundzie głosowania: sformułuj prompt na żywo, pokaż jak AI generuje kod.
🔹 Testuj po każdym kroku — uruchom aplikację i pokaż wynik.
🔹 Jeśli AI wygeneruje błąd — to moment dydaktyczny! „Dlatego nadal potrzebujemy ludzi."

Czas: ~20-25 minut (4 rundy × ~5 min)

🎨 Runda 1: Co budujemy?

Głosujcie reakcjami Zoom!

👍 Dashboard zgłoszeń IT — tabela, filtry, statusy

🎉 Wyszukiwarka bazy wiedzy — search bar + wyniki

😲 Monitor statusu systemów — karty, kolory, uptime

🔹 Daj 10 sekund na reakcje. Policz i ogłoś wynik.
🔹 Sformułuj prompt np.: „Zbuduj aplikację Streamlit: dashboard zgłoszeń IT z tabelą, filtrami po kategorii i statusie, i kolorowym priorytetem."
🔹 Uruchom AI — pokaż jak generuje kod.

🎨 Runda 2: Jaki framework?

Głosujcie!

👍 Streamlit — Python, prosty, idealny do prototypów

🎉 Flask + HTML — klasyka, pełna kontrola

😲 Czysty HTML/JS — zero backendu, od razu w przeglądarce

🔹 Dla tego typu aplikacji Streamlit jest najszybszy — ale niech głosują!
🔹 Dostosuj prompt do wybranego frameworka.
🔹 Po wygenerowaniu — uruchom i pokaż wynik na żywo.

🎨 Runda 3: Pierwsza funkcja?

Głosujcie!

👍 Wykresy i statystyki — ile zgłoszeń per kategoria

🎉 Wyszukiwanie — filtruj po tekście, kategorii

😲 Formularz dodawania — nowe zgłoszenie z poziomu UI

🔹 Prompt np.: „Dodaj wykres słupkowy z liczbą zgłoszeń per kategoria i wykres kołowy ze statusami."
🔹 Pokaż jak AI rozumie kontekst istniejącego kodu i dodaje nową funkcję.

🎨 Runda 4: Finishing touch?

Ostatnie głosowanie!

👍 Dark mode — ciemny motyw

🎉 Eksport CSV — pobierz dane

😲 AI klasyfikator — wpisz zgłoszenie, AI kategoryzuje

🎯 Aplikacja gotowa! Od zera do działającego narzędzia w ~20 minut.

🔹 Po ostatnim kroku podsumuj: „Właśnie zbudowaliśmy działającą aplikację w 20 minut. Żaden z Was nie pisał kodu — a mimo to wiecie, co się dzieje pod spodem, bo dzisiaj budowaliście agentów ręcznie. To jest prawdziwy power user 2026."

Puenta:
„Vibe coding to nie magia — to narzędzie. Działa najlepiej, gdy ROZUMIESZ co AI generuje. Dlatego najpierw uczymy się pisać agentów od zera, a potem przyspieszamy z AI."

🔹 Jeśli Ollama wolno odpowiada — wyjaśnij, że to mały model na CPU, w produkcji byłby szybszy na GPU lub przez API.

🤔 Co LLM umie – a czego nie?

  • ✅ Generowanie tekstu, kodu, podsumowań, tłumaczenie
  • ✅ Klasyfikacja, analiza sentymentu, ekstrakcja danych
  • ❌ Nie „rozumie" świata – halucynacje, wymyślanie faktów
  • ❌ Nie ma pamięci między sesjami (chyba że mu ją zbudujesz!)
  • ⚠️ Context Engineering rozwiązuje wiele tych ograniczeń

Intro:
„Zanim pójdziemy dalej — zatrzymajmy się na chwilę. LLM to potężne narzędzie, ale ma realne ograniczenia."

Storytelling:
🔹 „LLM to bardzo pewny siebie znajomy. Mówi pięknie, płynnie... ale czasem kompletnie wymyśla fakty. Dlatego potrzebujemy Context Engineering — żeby dać mu prawdziwe źródła."
🔹 „W jednej firmie prawnik użył wyników AI w sądzie. Problem? AI wymyśliło 3 z 5 cytowanych precedensów. To nie anegdota — to realny przypadek."

Aktywizacja:
🔹 Zapytaj: „Kto z Was miał sytuację, że AI pewnie podało błędną odpowiedź?"

🤖 Od modeli do agentów AI

LLM to potężny silnik… ale sam nie wie, co ma robić 🚗

  • Agent = LLM + Tools + Memory + Planning
  • Model decyduje, które narzędzie użyć
  • Pamięta kontekst rozmowy
  • Planuje kolejne kroki

🎯 To jest kierunek, w którym idzie cała branża AI w 2026

Intro:
„Do tej pory budowaliśmy chatboty — odpowiadały na pytania. Teraz budujemy agentów — wykonują zadania. LLM to silnik. Agent to cały samochód."

Storytelling:
🔹 „Chatbot to informacja. Agent to akcja. Chatbot mówi: 'Twoje zgłoszenie dotyczy sieci.' Agent mówi: 'Sprawdzam ping do drukarki... brak odpowiedzi... tworzę ticket w Jira... powiadamiam zespół sieciowy.' To fundamentalna zmiana."
🔹 „Model to jak nowy samochód bez kierowcy. Agent to kierowca, który wie, gdzie jechać, zatrzyma się na czerwonym i zamówi wam kawę po drodze."

📚 Komponenty agenta AI

  • 🧠 Model językowy (LLM): rozumienie instrukcji, podejmowanie decyzji
  • 🔧 Narzędzia (Tools): API, bazy danych, funkcje systemowe
  • 💾 Pamięć: krótkoterminowa (rozmowa) i długoterminowa (baza wiedzy)
  • 📋 Planowanie: podział zadania na kroki, ustalanie kolejności
  • 👁️ Monitorowanie: kontrola poprawności i bezpieczeństwa

Wyjaśnienie:
🔹 Pokaż 5 komponentów jako puzzle — każdy jest potrzebny do pełnego agenta.
🔹 LLM = mózg. Tools = ręce. Memory = notatnik. Planning = strategia. Monitoring = sumienie.

Storytelling:
🔹 „Przykład: agent logistyczny. Sprawdza API pogody → analizuje bazę zamówień → decyduje, czy zmienić trasę → wysyła powiadomienie kierowcy. Każdy krok to inny komponent."

🔀 Wzorce orkiestracji agentow

Nie kazdy agent to ta sama architektura. Oto 4 glowne wzorce:

🚦 Router

Klasyfikuj → wybierz sciezke → wykonaj

Analogia: recepcja hotelowa — goscie kierowani do wlasciwego pietra

Helpdesk, triage, routing

📋 Plan-and-Execute

Jeden model planuje, tani model wykonuje

Analogia: architekt + ekipa budowlana — plan rysuje drogi, buduje tani

Redukcja kosztow o 90%

🪞 Reflect

Agent sprawdza swoj wynik i poprawia

Analogia: redaktor prasowy — pisze, czyta, poprawia, czyta znowu

QA, code review, recenzje

👤 Human-in-the-Loop

Agent pyta czlowieka w kluczowych momentach

Analogia: pilot automatyczny — leci sam, ale laduje pilot

Finanse, medycyna, prawo

Intro:
„Nie kazdy agent to ta sama architektura. W zaleznosci od problemu, wybieramy inny wzorzec. Po obiedzie zbudujemy Router w LangGraph — ale warto wiedziec, ze sa tez inne."

Storytelling:
🔹 Router = recepcja hotelowa — gosc mowi czego potrzebuje, recepcjonista kieruje na wlasciwe pietro.
🔹 Plan-and-Execute = architekt rysuje plan, ekipa budowlana wykonuje. Architekta placi za myslenie, ekipe za rece.
🔹 Reflect = redaktor, ktory pisze artyktul, czyta go krytycznie, poprawia, czyta znowu. Petla samopoprawy.
🔹 Human-in-the-Loop = autopilot w samolocie — leci sam, ale ladowanie robi pilot.

Aktywizacja:
🔹 „Ktory wzorzec pasowałby do waszego agenta z wczesniejszego cwiczenia?"

📝 Ćwiczenie: Zaprojektuj swojego agenta

W grupach 2-3 osoby – 10 minut:

  1. Wybierz problem z pracy, który agent mógłby rozwiązać
  2. Narysuj: Input → Analiza → Decyzja → Akcja
  3. Określ: narzędzia? Pamięć? Reguły?
  4. Przedstaw w 1 minucie na czacie
Przykład: Agent HR — CV (input) → analiza doświadczenia (LLM) → wymagania (tool: baza HR) → zaproszenie/odrzucenie → mail (action)

Intro:
„Czas na waszą kreatywność! Zaprojektujcie agenta, który rozwiązuje realny problem z waszej pracy."

Jak poprowadzić:
🔹 Grupy 2-3 osoby (breakout rooms w Zoomie lub przy stolikach).
🔹 Zadanie: wybierz problem → narysuj schemat → określ narzędzia i reguły.
🔹 10 min projektowanie + 5 min prezentacje (1 min na grupę).
🔹 Jeśli zdalnie: poproś o wpisanie schematu tekstowo na czacie.

Wskazówki do moderowania:
🔹 Jeśli ktoś nie wie, co wymyślić: podaj przykład agenta HR — CV → analiza doświadczenia → sprawdzenie wymagań → zaproszenie/odrzucenie → mail.
🔹 Obserwuj grupy — podpowiadaj tym, które utknęły.
🔹 Pochwal kreatywne pomysły!
